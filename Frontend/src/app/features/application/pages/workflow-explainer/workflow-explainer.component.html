Workflow Creation: How It Is Saved

Step-as-model architecture (current implementation)

1. What You Design In Builder

Step

Each step contains its own schema in fields[]. There is no separate model mapping now.

Edge

Edges define routing and permission rules: target step, terminal, allowed roles, submitter-only.

Data Config

Reference previous step, reuse previous record, auto field mapping, read-only fields.

2. Save Draft / Publish Flow

Builder JSON
↓
POST /workflows (DRAFT)
↓
Mongo: workflow_definitions
↓
Publish Validation
↓
PUBLISHED Version

Validation checks include: start step, reachable graph, edge permissions, required fields, mapping/reuse consistency.

3. Runtime: When Instance Is Actually Saved

Start Screen

Opening start screen does not create an instance.

Start Submit

Instance is created only when user submits first step from the start flow.

Execute Edge

Current step data is validated against step.fields and stored in instance.stepRecords[stepId].

Tracking

History records each transition. Tasks are generated by edge permissions for current user roles.
