Workflow Builder Guide

Everything you need to build a process from scratch

How a Workflow Works

A workflow is a sequence of nodes connected by edges. When a user starts the process, it advances through each node in order, pausing at interactive steps until the user acts.

START
→
FORM PAGE
→
DATA ACTION
→
APPROVAL
→
END

Step-by-step to build a workflow

  1. Go to the Nodes tab — add a START node first.
  2. Add your interactive nodes: FORM_PAGE, APPROVAL, DATA_VIEW.
  3. Add automatic nodes: DATA_ACTION, CONDITION, NOTIFICATION.
  4. Finish with an END node.
  5. Go to the Edges tab — connect every node in order.
  6. Go to Settings — configure auth and draft options.
  7. Click Save Draft to save, then Publish to make it live.
Every process must have exactly one START and one END node. All other nodes must be reachable from START via edges.

Node Types

Each node type has a specific role. Choose the right type for each step in your workflow.

START

The entry point. Every workflow begins here. Add exactly one. No configuration needed — the engine auto-advances past it.

END

The exit point. When the process reaches END, the instance is marked Completed. Add exactly one.

FORM_PAGE

Shows a form to the user. The process pauses here until the user fills and submits it. You configure:

  • Elements — input fields (text, number, date, select, radio, checkbox, etc.)
  • Submit label — text on the submit button (default: Submit)
  • Visibility rules — show/hide a field based on another field's value
  • Required — mark fields as mandatory

Data collected here is referenced in later nodes as nodeId__fieldId.

APPROVAL

Presents collected process data to an approver who must take an action. Configure:

  • Actions — e.g. Approve / Reject (you define the labels and IDs)
  • Comment — approver can optionally leave a note

Connect different edges for each action using a CONDITION node or edge conditions after this node.

DATA_ACTION

Runs automatically — no user interaction. Writes data to a linked model. Configure:

  • Model — which data model to write to
  • Operation — CREATE, UPDATE, or DELETE
  • Field mappings — map model fields from form data (FORM_FIELD) or a fixed value (STATIC_VALUE). Use the dropdown to pick fields collected in ancestor FORM_PAGE nodes.
DATA_VIEW

Shows records from a linked model to the user (read-only table). The process pauses until the user clicks Continue. Configure:

  • Model — which model to query
  • Filter field — optionally filter rows by a form field value
CONDITION

Runs automatically. Evaluates one or more rules and routes to different edges. Configure:

  • Rules — each rule has a field, an operator, a value, and a target edge ID
  • The first matching rule wins. If none match, the default (no-label) edge is used.
NOTIFICATION

Sends an email/notification automatically. Configure:

  • Recipient — email address or a form field reference
  • Subject & Body — message content

Edges — Connecting Nodes

An edge is a directed connection from one node to another. The workflow follows edges to determine what happens next.

How to add an edge
  1. Go to the Edges tab.
  2. Click Add Edge.
  3. Set From Node — where the edge starts.
  4. Set To Node — where the edge leads.
  5. Optionally set a Label (e.g. "approved", "rejected") — used by CONDITION nodes to route to the right edge.
  6. Optionally set a Condition — a simple field/operator/value check. If this edge has a condition, it only activates when the condition is true.
Rules for edges
  • Every node except END must have at least one outgoing edge.
  • Every node except START must have at least one incoming edge.
  • CONDITION nodes need one edge per rule — each edge has a different label matching the rule's target edge ID.
  • APPROVAL nodes typically have one edge per action (approve/reject).
  • A node with multiple outgoing edges should use a CONDITION node to decide which path to follow.
The Flow Preview tab shows a visual diagram of all nodes and edges. Nodes with a red border are not reachable from START — add edges to connect them.
Edge label vs edge condition
PropertyPurposeWhen to use
LabelNamed route that a CONDITION node targets by IDWhen using a CONDITION node to split paths
ConditionField/operator/value check evaluated at runtimeWhen you want a direct rule on the edge itself without a CONDITION node

Conditions & Branching

Conditions let your workflow take different paths depending on data collected from forms or approval actions.

Method 1 — CONDITION Node

Add a CONDITION node after an APPROVAL or FORM_PAGE. In its config add one rule per possible path:

Fieldapproval_action
OperatorEQUALS
Valueapprove
Target Edgeapproved-path (the label of the edge that leads to the next step)

Connect the CONDITION node to two different nodes with two edges — one labeled "approved-path", one labeled "rejected-path". The engine routes to the matching edge.

Method 2 — Edge Condition (direct)

On any edge you can set an inline condition. The engine will only follow that edge if the condition is true. If multiple edges have conditions, the first matching one wins.

Operators reference
OperatorMeaning
EQUALSField value exactly matches the given value
NOT_EQUALSField value does not match
GREATER_THANNumeric field is greater than value
LESS_THANNumeric field is less than value
CONTAINSText field contains the value as a substring
IS_EMPTYField has no value
IS_NOT_EMPTYField has any value
Field reference format

When referencing a field collected in a FORM_PAGE, use the format:

nodeId__fieldId

In DATA_ACTION field mappings, use the dropdown to select from available ancestor fields — the system builds this reference automatically.

Tips, Rules & Common Mistakes

Required rules
  • Exactly one START and one END node per process.
  • All nodes must be reachable from START — check the Flow Preview tab for red-bordered disconnected nodes.
  • Every node must have at least one outgoing edge (except END).
  • A process must be in DRAFT status to edit. Publish to make it live.
  • Once published, the process is locked. Archive it first, then create a new draft to modify.
DATA_ACTION tips
  • Link at least one model before building DATA_ACTION nodes — go to Linked Models at the top.
  • For FORM_FIELD source: use the dropdown — it shows fields from ancestor FORM_PAGE nodes only. Fields from nodes that aren't predecessors in the flow won't appear.
  • For STATIC_VALUE source: type a fixed value (e.g. "pending", "true", "42").
  • Unmapped model fields will be stored as null.
Form element tips
  • Give every element a unique ID — it is used as the reference key in conditions and DATA_ACTION mappings.
  • Use Visibility Rules to create conditional form fields — e.g. show an "Other" text input only when the user selects "Other" from a dropdown.
  • Import fields from a linked model to auto-generate a form matching your data schema.
Common mistakes
MistakeFix
Process won't publish — validation errorsCheck that START and END exist, all nodes have edges, and required configs are set
DATA_VIEW shows — (dashes)The DATA_ACTION that saves records ran with empty field mappings. Reconfigure mappings and re-run the process
CONDITION node doesn't route correctlyMake sure the Target Edge ID in the rule exactly matches the Label on the outgoing edge
