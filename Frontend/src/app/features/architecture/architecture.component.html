1. System Overview

AdaptiveBP is a multi-tenant business process platform. Each tenant is a Domain (organisation). Within a domain, owners create Applications. Each application has one published Process Definition — a directed graph of nodes. End users run Process Instances that walk through that graph step by step, collecting form data, triggering approvals, and querying data models.

2. Entity Hierarchy

All entities are stored in MongoDB. Each level scopes the one below it.

Owner Account
Single global admin. Created at platform setup. Owns all domains.
Domain (Organisation)
Workspace. Has slug, industry, theme, domain groups and members. Many per Owner.
Domain Group
Grants domain-level permissions (DOMAIN_MANAGE_APPS, DOMAIN_USE_APP…). Users are members.
Application
A scoped workflow product. Has slug, app groups, data models. Many per Domain.
App Group
Grants app-level permissions (APP_READ, APP_WRITE, APP_EXECUTE, APP_START_PROCESS…).
Domain Model (Schema)
Field schema definition attached to an application. Used by DATA_ACTION nodes to write records.
Model Record
Actual data rows. Stored as Map<String,Object>. Written by DATA_ACTION, read by DATA_VIEW.
Process Definition
Node/edge graph. Versioned. One PUBLISHED per application at any time.
Process Instance
Runtime copy of a definition. Tracks currentNodeId, status, history, collected data.

3. Access Control

Access is enforced at four levels. No role-based annotations — all checks are manual in the PermissionService called from each controller.

L1
Owner

PrincipalType = OWNER. Bypasses every permission check. Created via owner-signup, authenticated with a separate JWT.

L2
Domain Group Member

PrincipalType = DOMAIN_USER. Permissions come from domain groups the user belongs to. Checked for domain-scoped operations (create app, manage users).

L3
App Group Member

Same DOMAIN_USER, but permissions resolved from app groups. APP_WRITE = access Process Builder + Data Models. APP_EXECUTE = Access Manager. Any authenticated user can start and run.

L4
Authenticated (no group)

Any logged-in DOMAIN_USER can start a process instance, view their own instance, and submit steps. Cannot see instance list, process builder, or data models.

Node types

Field reference format

When a DATA_ACTION node maps a field from a prior FORM_PAGE, the reference is encoded as nodeId__elementId (double underscore, avoiding MongoDB dot notation). The engine resolves this at runtime by reading instance.collectedData[nodeId][elementId].

Process definition graph rules

  • Every definition must have exactly one START node and at least one END node.
  • Edges are directed: fromNodeId → toNodeId. Optional label (e.g. "approve", "reject") used by APPROVAL routing.
  • CONDITION nodes evaluate a boolean and follow the edge whose label matches "true" or "false".
  • Cycles are not supported — the engine has a depth guard (max 50 auto-advance hops) to prevent infinite loops.
  • Only one definition can have status PUBLISHED at a time per application. Publishing a new version archives the previous one.

7. Tech Stack

Frontend
  • Angular 17 — Module-based routing, *ngIf structural directives
  • TypeScript — Strict typing, interface-first models
  • CSS Custom Properties — Theme vars (--tc, --tp, --ta) injected per-component
  • LocalStorage — Theme persistence (at-[domain]-[app], dt-[domain])
  • Reactive HTTP — Angular HttpClient with Observable chains
Backend
  • Spring Boot 3 — REST controllers, Jackson serialisation
  • Spring Security — Stateless JWT filter, SecurityContextHolder
  • MongoDB — Document store via Spring Data MongoDB
  • Port/Adapter pattern — Controllers call service ports, not repositories directly
  • BCrypt — Password hashing for domain users
Infrastructure
  • Docker / Docker Compose — Backend, MongoDB, Frontend containers
  • GitHub Actions CI — Build + test pipeline on push
  • Port 8005 — Backend API default port
  • Port 4200 — Angular dev server
